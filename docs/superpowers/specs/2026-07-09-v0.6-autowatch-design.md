# meshwire v0.6 — auto-watch after init/join

**Date:** 2026-07-09
**Status:** approved design, pre-implementation
**Target version:** 0.6.0 (mesh.py stays a single stdlib-only file)

## Goal

One command per machine: after `mesh init` or `mesh join` succeeds in a
terminal, the process flows directly into `watch --follow`. Closing the
program (Ctrl-C, closing the terminal) stops the watcher — no separate
watch command for humans.

## Behavior

- After a successful `cmd_init` or `cmd_join` (config written, identity
  set, announce sent for join), the command checks `_interactive()` — a
  helper returning `sys.stdout.isatty()`.
- **Terminal (isatty true):** print one line —
  `listening for messages — Ctrl-C to stop` — then run the follow-watcher
  in-process (call `cmd_watch` with `follow=True, timeout=None,
  as_node=None`). The process never returns to the prompt until
  interrupted. Ctrl-C exits cleanly: `main()` wraps command dispatch in a
  `KeyboardInterrupt` handler that exits 130 (conventional SIGINT code) without a traceback (this also
  fixes Ctrl-C on a manually run `mesh watch --follow`).
- **Not a terminal (scripts, tests, cron, Claude Code bash):** behavior
  identical to v0.5 — the command returns immediately after setup. The
  caller owns watching. No flags; the distinction is automatic and this is
  the whole mechanism (no `--watch`/`--no-watch` options).
- The decision lives in a small helper (`_interactive()`) so tests can mock
  it for both paths.

## Ripples

- **Invite paste block** (`cmd_invite` output) drops its
  `&& python3 mesh.py watch --follow` tail — join listens by itself when
  pasted into a terminal. The `--as <name>` and pipx alternative lines
  stay.
- **`cmd_init` output**: in a terminal, init prints the full invite block
  itself (extracted helper `_print_invite(cfg)`, shared with `cmd_invite`)
  before flowing into the watcher — the user cannot run `mesh invite` in a
  terminal that is now busy watching. Non-TTY init keeps today's "run
  `mesh invite`" pointer line. The "listen here: `mesh watch --follow`"
  hint is dropped in both modes.
- **README**: quick start becomes one command per machine; Machine A's
  block drops the explicit `mesh watch --follow &` line (init itself
  listens, so the `MESH_NODE_JOINED` promise still holds); "Using it with
  Claude Code" and the CLI reference gain one sentence: in a terminal,
  init/join keep running as the watcher; programs calling mesh (non-TTY)
  get the old return-immediately behavior.
- **SKILL.md / CLAUDE_SNIPPET / AGENTS.md**: protocol unchanged (agent
  shells are non-TTY). Add one clarifying sentence to SKILL.md only, so a
  session isn't confused if a user reports "init kept running" — no
  protocol edits.
- **Ctrl-C during the auto-watch** must not roll back or dirty anything:
  setup completed before watching began; the cursor file semantics are the
  watcher's normal ones.

## Testing

- Unit tests (stdlib, no network, appended to tests/test_mesh.py):
  - init with `_interactive` mocked true → `cmd_watch` called with
    follow=True (mock `cmd_watch`, assert args) after config exists.
  - init with `_interactive` mocked false → `cmd_watch` not called
    (current behavior).
  - join with `_interactive` true → announce sent before watching starts
    (order), `cmd_watch` called.
  - KeyboardInterrupt from the auto-watch exits 0 without traceback.
  - Invite block no longer contains `watch --follow`.
- Existing tests keep passing unmodified: under unittest, captured stdout
  is not a TTY, so `_interactive()` is false by default.

## Compatibility

- Wire format, config, join codes: untouched.
- v0.5 behavior preserved exactly for every non-TTY caller.
- Version bumps: pyproject.toml and .claude-plugin/plugin.json → 0.6.0,
  `USER_AGENT` → `meshwire/0.6`.

## Non-goals

- No `--watch`/`--no-watch` flags (Ctrl-C covers the human opt-out;
  non-TTY covers automation).
- No daemonizing/backgrounding of the watcher by init/join.
