# meshwire v0.5 — "Frictionless mesh" design

**Date:** 2026-07-09
**Status:** approved design, pre-implementation
**Target version:** 0.5.0 (mesh.py stays a single stdlib-only file)

## Goals

1. **Easier onboarding.** No pre-declared node list; a machine holding the
   join code joins with one pasted command. Better invite ergonomics.
2. **Faster ask/reply.** The transport is already ~1s; the latency lives in
   the one-shot watcher's re-arm gap and per-send TLS handshakes. Close both.
3. **Clear README.** Quick start first, everything else below the fold.

Non-goals (explicitly deferred): MCP server (`mesh mcp-serve` for shell-less
clients like Claude Desktop — future), resident daemon / OS services, file
transfer, relay-hosted short invite codes.

## 1. Dynamic membership

### init / join / iam

- `mesh init <name> [--as NODE] [--nodes a,b] [--server URL]` — `--nodes` is
  now **optional** (accepted for back-compat; seeds the known list). Config
  `nodes` is seeded with this machine's identity.
- `mesh join <code> [--as NODE]` — unchanged semantics, but afterwards the
  machine **broadcasts an announce** control message (see §3) so peers learn
  it immediately.
- Identity defaults to the **sanitized hostname** when `--as` is omitted, in
  both `init` and `join`: `socket.gethostname()`, lowercased, trailing
  `.local`/`.lan` stripped, non-`[a-z0-9-]` → `-`, runs collapsed, edges
  trimmed. If the result is empty or equals the reserved name `all`, an
  explicit `--as` is required. The chosen name is always printed.
- `mesh iam <node>` accepts **any** name except `all`; if the name is not in
  `nodes` it is added and the config saved.
- `my_node()` no longer errors when the identity is missing from `nodes`; it
  auto-adds and saves.

### Learning peers

- **Any authenticated inbound message from an unknown sender registers that
  sender**: appended to config `nodes`, and recorded in a new per-machine,
  gitignored file `.meshwire.peers.json` mapping
  `node → {"seen": epoch, "via": "message"|"announce"|"pong"}`.
- A shared helper (`note_peer(cfg, node, via)`) is called from every inbound
  path: `watch` (both modes), `peek`, `_await_result`, and pong handling.
- `mesh status` lists known peers with human-readable last-seen ("2m ago",
  "never").
- `.meshwire.peers.json` joins the auto-gitignore list written by
  `_write_config_here`.

### Sending

- `mesh send` / `mesh ask` to a name not in `nodes`: **warn on stderr**
  ("never seen '<name>' — sending anyway") and proceed. Topics are derived
  purely from `mw-<mesh>-<id>-<node>`, so delivery needs no registration;
  the warning exists only to catch typos. Sending to self remains an error.

## 2. Invites

`mesh invite` prints, in addition to today's join code line, a complete
paste-able bootstrap block for the new machine:

```bash
curl -fsSLO https://raw.githubusercontent.com/husker/meshwire/main/mesh.py
python3 mesh.py join mesh1-XXXX... && python3 mesh.py watch --follow
```

One paste = downloaded, joined, named (hostname default), listening. A
pipx/uv alternative is mentioned beneath it. `mesh init` output points at
`mesh invite` instead of duplicating the block.

## 3. Control messages (announce / ping / pong)

Wire format: the encrypted wrapper gains an optional `"c"` field alongside
the human-readable `"b"` body:

```json
{"f": "desktop", "t": "all",    "b": "desktop joined the mesh", "c": {"mw": "announce"}}
{"f": "laptop",  "t": "desktop","b": "ping",                    "c": {"mw": "ping", "n": "<nonce>", "ts": 1234.5}}
{"f": "desktop", "t": "laptop", "b": "pong",                    "c": {"mw": "pong", "n": "<nonce>", "ts": 1234.5}}
```

Behavior in v0.5 clients (both watch modes and `peek`):

- Control messages are **never printed as `MESH_MESSAGE`** and never consume
  a one-shot watch. They update the peers file and are logged to stderr.
- **Announce** additionally prints `MESH_NODE_JOINED node=<name>` to stdout
  (rare, informative, agent-safe marker).
- **Ping** triggers an immediate automatic pong (echoing the nonce) from any
  running watcher — one-shot or `--follow` — without agent involvement.

Mixed-version caveat (documented in README): v0.4 clients render control
messages as ordinary `MESH_MESSAGE` lines with the `"b"` text ("desktop
joined the mesh", "ping") — harmless but noisy; upgrade all machines
together (it is one file).

## 4. Speed

### `mesh watch --follow`

- Persistent mode: holds the streaming connection, prints each trusted
  message (same `MESH_MESSAGE`/`MESH_TASK` + JSON lines as one-shot), flushes
  per event, and keeps listening. Runs forever unless `--timeout N` is given.
- Cursor is persisted after **every** event (exactly the one-shot logic), so
  kill/crash/restart neither loses nor replays messages.
- Reconnects on network errors with backoff 1s → 2s → … capped at 30s.
- Default `mesh watch` stays one-shot; nothing existing breaks. The plugin
  skill, `claude-setup` output, and README teach `--follow` (under Claude
  Code's Monitor tool) as the preferred mode.

### `mesh ping <node> [--timeout N]` (default 30)

Opens own inbox stream, sends the ping control message, awaits the matching
pong, prints `MESH_PONG node=<x> rtt=<ms>`; on timeout prints a hint ("no
watcher running on <x>, or offline") and exits nonzero. Updates peers.

### Wire-level fixes

- `mesh ask --wait`: the result stream is **established before** the ask is
  transmitted (today it sends first, then connects; a fast reply can add a
  reconnect/replay delay).
- `send_raw` reuses a module-level `http.client.HTTPSConnection` per host
  (keep-alive) instead of a fresh TLS handshake per send (~0.3–0.5s each);
  on any connection error it reconnects once and retries. Streaming reads
  keep using urllib as today.

Expected outcome: transport round trip ~1–2s. Remaining ask latency is the
far agent thinking — now diagnosable (fast `ping` + slow reply ⇒ agent, not
mesh).

## 7. Testing

- **Unit tests** (new `tests/test_mesh.py`, stdlib `unittest`, no network;
  transport stubbed by monkeypatching `mesh.http`): crypto round-trip and
  tamper rejection, join-code round-trip, hostname sanitization, envelope
  parsing, peer learning (unknown sender → nodes + peers file), control
  routing (ping → auto-pong; announce → MESH_NODE_JOINED; neither consumes
  one-shot watch), send-to-unknown warning.
- **Live smoke** (manual checklist against real ntfy.sh, two temp-dir
  configs): join + announce learning; send → `--follow` delivery; `ping`
  RTT < 3s; scripted ask → auto-reply round trip < 5s. Results feed the
  README's latency claims.

## 8. Compatibility summary

- Config shape and join-code format unchanged; v0.4 meshes keep working.
- New files: `.meshwire.peers.json` (per-machine, gitignored).
- Old clients: can't send to unlisted nodes, and render control messages as
  odd one-off `MESH_MESSAGE`s → README: upgrade every machine together.
